# 2. Определение дерева метрик и SLO для ML-сервиса онлайн-кинотеатра

Дата: 2026-05-17

## Статус

Accepted

## Контекст

ML-сервис разрабатывается для рекомендательной системы онлайн-кинотеатра. Перед началом разработки разные команды сформулировали требования, которые необходимо согласовать между собой 
(некоторые из них конфликтуют по условию).

Бизнес хочет увеличить выручку онлайн-кинотеатра, выдерживать пиковую нагрузку 10000 RPS, получить быстрый и надежный сервис.

Команда разработки обсуждает необходимость нового сервера, перехода на ClickHouse, переписывания backend и возможного перехода на Kubernetes.

ML команда обсуждает необходимость GPU, выбор числа фильмов в рекомендательной выдаче, переход на stream-processing и использование KubeFlow.

DevOps команда указывает, что существующие мощности могут быть недогружены, PostgreSQL в текущей конфигурации показывает только 1000 RPS, а SLA 100% является нереалистичной целью.

Поэтому перед архитектурными изменениями нужно определить сбалансированное дерево метрик, которое позволит принимать решения на основании измерений, а не предположений.

## Решение

Мы выбираем четыре группы метрик:

1. бизнес-метрики;
2. метрики приложения;
3. ML-метрики;
4. инфраструтурные метрики.

### Бизнес-метрики

- Revenue / ARPU — бизнес-метрика выручки на пользователя;
- Recommendation CTR — доля кликов по рекомендациям;
- Conversion to view — доля случаев, когда рекомендация привела к просмотру фильма;
- Retention — удержание пользователей.

Эти метрики отвечают на требование бизнеса увеличить выручку и проверить, действительно ли рекомендации улучшают пользовательское поведение.

### Метрики приложения

- RPS, включая ориентир на пиковую нагрузку 10000 RPS — requests per second, количество запросов к сервису в секунду;
- p95 latency — 95-й перцентиль времени ответа сервиса;
- error rate — доля ошибочных ответов сервиса;
- availability — доля времени, когда сервис доступен пользователям;
- timeout rate — доля запросов, завершившихся по таймауту.

Эти метрики отвечают на требования скорости, надежности и способности сервиса выдерживать пиковую нагрузку.

### ML-метрики

- Precision@K / Recall@K / NDCG@K — стандартные метрики качества рекомендательных систем;
- Quality by K — зависимость качества рекомендаций от числа фильмов, показываемых пользователю;
- inference latency — время, необходимое модели для построения рекомендаций;
- data drift / prediction drift — смещение распределения входных данных или распределения предсказаний модели.

Эти метрики позволяют оценить качество рекомендательной модели, выбрать разумное число фильмов в выдаче, а также понять, нужна ли оптимизация inference, GPU, stream-processing или KubeFlow.

### Инфраструктурные метрики

- CPU / RAM utilization — степень использования процессора и оперативной памяти;
- GPU utilization — степень использования GPU при обучении или inference модели;
- PostgreSQL QPS — количество запросов к PostgreSQL в секунду;
- DB query latency — время выполнения запросов к базе данных;
- container restarts — количество перезапусков контейнеров сервиса.

Эти метрики позволяют проверить, действительно ли существующие мощности недогружены, где находится узкое место системы и нужны ли новый сервер, Kubernetes или изменение конфигурации инфраструктуры.

Для первой итерации мониторинга принимаем следующие SLO:

```text
p95 latency < 1 секунда
error rate < 1%
availability > 99%
```

Основным SLO для настройки алерта выбираем:

```text
p95 latency < 1 секунда
```

Алерт срабатывает при условии:

```text
p95 latency > 1 секунда в течение 2 минут
```

## Обоснование

Latency p95 выбран как основной SLO, потому что он напрямую связан с требованием бизнеса «чтобы быстрый был», с вопросом DevOps о требуемом времени отклика и с техническими решениями разработки: нужен ли новый сервер, переписывание backend или изменение хранилища.

RPS нужен для проверки требования о пиковой нагрузке 10000 RPS.

Error rate и availability нужны для проверки требования надежности.

ML-метрики нужны, чтобы определить качество рекомендаций и выбрать число фильмов в выдаче. Метрика Quality by K показывает, как меняется качество рекомендаций при разных значениях K, поэтому она лучше формулирует задачу выбора числа фильмов, чем просто параметр Optimal K.

Инфраструктурные метрики нужны, чтобы проверить позицию DevOps о недогруженных мощностях и определить, где находится bottleneck: в CPU/RAM/GPU, PostgreSQL, контейнерах или коде сервиса.

## Последствия

Команды получают общий язык для обсуждения архитектурных решений.

Новый сервер, GPU, ClickHouse, Kubernetes или KubeFlow не выбираются заранее. Эти решения принимаются только после анализа метрик.

Первая реализация мониторинга фокусируется на p95 latency, потому что эту метрику легко собрать через Prometheus, вывести в Grafana и нарушить искусственной задержкой для проверки алерта.

## Риски

- Хорошие ML-метрики не гарантируют рост выручки.
- Увеличение числа фильмов K может улучшить Recall@K, но ухудшить latency и пользовательский опыт.
- GPU может быть не нужен, если bottleneck находится в backend или PostgreSQL.
- ClickHouse может не решить проблему, если задержка возникает на inference модели.
- Kubernetes может усложнить систему без реальной необходимости.
- Data drift может ухудшить качество рекомендаций без роста error rate.
- SLA 100% не используется, потому что он практически недостижим.

## Связанные артефакты

Дерево метрик строится кодом в ноутбуке и сохраняется в папку `screenshots/`.
Основная Prometheus-метрика:

```text
request_latency_seconds
```

PromQL-запрос для p95 latency:

```promql
histogram_quantile(0.95, sum(rate(request_latency_seconds_bucket[5m])) by (le))
```
